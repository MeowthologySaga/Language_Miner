# AI Build Workflow

AI는 아이디어를 바로 코드로 옮기지 말고, 게임 구조와 다이아 흐름을 먼저 고정합니다.

## 1. 입력 분석

사용자 입력 예시:

```md
고양이들이 우주 정거장에서 생선을 모으고 시설을 확장하는 게임을 만들고 싶어.
```

AI는 먼저 다음을 추론합니다.

- 후보 장르
- 핵심 행동
- 성장 루프
- 다이아 소비 가능 지점
- 필요한 에셋
- 적합한 엔진

## 2. 제작 모드 질문 게이트

AI는 구현 전에 반드시 제작 모드를 질문하고 답변을 기다립니다.

다음 표현은 구현 허가가 아니라 아이디어 입력으로만 해석합니다.

- "뱀서류 만들자"
- "방치형 게임 만들어줘"
- "알아서 만들어"
- "빠르게 해줘"
- "바로 만들어줘"

질문을 생략할 수 있는 경우는 사용자가 명시적으로 다음 중 하나를 말한 경우뿐입니다.

- "질문 없이 바로 만들어"
- "빠른 제작 모드로 진행해"
- "모든 선택은 AI가 알아서 하고 구현해"
- "제작 모드 질문 생략"

제작 모드가 확정되기 전에는 파일 생성, 코드 작성, 패키징을 하지 않습니다.

## 3. 장르 레시피 선택

기본 선택:

- 2D 게임: Phaser
- 3D가 반드시 필요한 게임: Three.js
- 텍스트/관리 위주: DOM + Canvas 또는 Phaser

AI는 3D가 꼭 필요하지 않으면 Phaser를 우선합니다.

## 4. Game Design Contract

구현 전 다음 형식으로 설계를 고정합니다.

```md
# Game Design Contract

## 한 줄 설명

## 핵심 재미

## 핵심 루프

## 1분 경험

## 5분 경험

## 장기 목표

## 실패/정체 구조

## 주요 시스템

## 에셋 방향

## 넣지 않을 것
```

## 5. Diamond Economy Contract

```md

## 다이아 소비처

## manifest economy.diamondActions 초안
```

## 6. Visual Approval Gate

구현 전에는 반드시 비주얼 검토 보드를 생성하고 사용자 승인을 받습니다.

필수 산출물:

- 인게임 느낌 보드: 핵심 플레이 화면, HUD, 액션, 보상 감각을 한 장으로 보여줍니다.
- 씬 플로우 보드: 메인 메뉴부터 결과/성장/재도전까지 주요 화면 전이를 한 장으로 보여줍니다.

씬이 거의 없는 아주 단순한 게임은 인게임 느낌 보드 1장만 허용합니다.
메뉴, 선택, 전투, 결과, 성장 화면이 있는 게임은 씬 플로우 보드를 반드시 생성합니다.

AI는 보드 생성 후 반드시 다음 질문을 합니다.

```md
이 이미지 방향이 정말 원하는 게임이 맞나요?
수정할 분위기, 화면 흐름, UI, 보상 구조가 있으면 말해주세요.
확인되면 이 방향으로 구현을 시작하겠습니다.
```

사용자가 승인하기 전에는 구현하지 않습니다.

## 구현 금지 조건

다음이 확정되기 전에는 구현을 시작하지 않습니다.

- 제작 모드
- 장르 레시피
- Game Design Contract
- Diamond Economy Contract
- 비주얼 검토 보드
- 사용자 비주얼 승인

단, 사용자가 명시적으로 "질문 없이 바로 만들어" 또는 "빠른 제작 모드로 진행해"라고 한 경우에만 빠른 제작 모드로 간주하고 최소 질문으로 진행할 수 있습니다.

## 7. 구현

AI는 다음 순서로 구현합니다.

```txt
1. 프로젝트 스캐폴드
2. 런처 API mock
3. 핵심 게임 루프
4. 저장
5. 다이아 소비 요청
6. 상점/BM UI
7. 에셋 적용
8. 설정/도움말
9. 런처 manifest
10. README
```

UI 스프라이트를 만드는 경우, 크기가 달라지는 버튼/패널/칩/슬롯은 9-slice 규칙을 우선 적용합니다. 텍스트가 들어간 버튼 이미지를 통째로 늘려 쓰지 말고, 프레임 스프라이트와 실제 텍스트 렌더링을 분리합니다.

## 8. 검증

AI는 최소한 다음을 확인합니다.

- PC 1280x720에서 플레이 가능
- 첫 보상 발생
- 저장/불러오기 작동
- 다이아 소비 요청이 mock에서 작동
- manifest가 유효함
- README가 존재함
- 외부 네트워크 의존이 없음
- `.lem`에 실행 불필요 파일이 과하게 들어가지 않음
- 주요 버튼/패널/칩 UI가 1280x720과 좁은 비율에서 깨지지 않음
